Selenium WebDriver Foundations:
A Complete Guide for Automation Engineers

This document consolidates all essential Selenium WebDriver concepts, from architecture and environment setup to writing and executing cross‑browser tests. Designed for Automation Engineers, Testers, and Developers who want a solid, practical understanding of Selenium WebDriver with modern best practices.

1. What is Selenium WebDriver?

Understanding WebDriver requires viewing it from three perspectives. These are also frequent interview topics.

1.1 WebDriver is a Component of the Selenium Suite

The Selenium ecosystem consists of three main components:

  • Selenium IDE – a record‑and‑playback tool for quick prototyping.
  • Selenium WebDriver – the core automation engine that directly controls the browser.
  • Selenium Grid – allows parallel and distributed test execution across multiple machines and browsers.

WebDriver is the component you will use for all real‑world test automation.

1.2 WebDriver is a Java Interface (Technical Foundation)

In Java, an interface is a blueprint that contains method signatures without implementation. The WebDriver interface defines all standard automation methods such as get(), close(), getTitle(), findElement(), etc.

Each browser has its own driver class that implements the WebDriver interface. Because all browser‑specific classes follow the same contract, method names and purposes are identical across Chrome, Firefox, Edge, Safari, and others.

The true architecture (Selenium 4.x, corrected):

SearchContext          (interface – root; provides findElement / findElements)
   ↑
WebDriver             (interface – extends SearchContext; defines browser‑control methods)
   ↑
RemoteWebDriver       (class – implements WebDriver; the base for all browser drivers)
   ↑
   ├── ChromiumDriver (class – common base for Chromium‑based browsers)
   │        ↑
   │   ┌────┴─────┐
   │ ChromeDriver  EdgeDriver
   │
   ├── FirefoxDriver (class – extends RemoteWebDriver directly)
   ├── SafariDriver   (class – extends RemoteWebDriver directly)
   └── ... (other browsers)

This design allows a single WebDriver variable to hold any browser’s driver object (polymorphism). Example:

WebDriver driver = new ChromeDriver();   // works
WebDriver driver = new FirefoxDriver();  // works
WebDriver driver = new EdgeDriver();     // works

The code you write remains browser‑agnostic.

1.3 WebDriver is an API (Application Programming Interface)

An API acts as an intermediary that receives requests from a client, processes them, and returns a response. In automation context, the WebDriver library acts as the API between your Java test code (client) and the browser (execution target). You call driver.get("url"); WebDriver translates that command into a browser‑specific protocol (now the W3C WebDriver standard) and instructs the browser. The browser’s response (e.g., current URL or page title) is sent back to your code.

Why this matters for you: WebDriver is not just a tool; it’s an interface that guarantees a consistent programming model, and the API nature means you can swap browsers without changing your test logic.

2. Environment Setup – The Right Way with Maven

As an automation engineer, you should avoid manual JAR management. Maven handles dependency resolution, version upgrades, and transitive dependencies automatically.

2.1 Creating a Maven Project in Eclipse

  • Launch Eclipse and choose/create a workspace (e.g., C:\Automation\SeleniumWebDriver).
  • Create a Maven project: File → New → Other → Maven → Maven Project
  • Check “Create a simple project (skip archetype selection)”.
  • Fill in: Group Id: e.g., com.automation, Artifact Id: e.g., selenium-webdriver.
  • Project structure: src/main/java, src/main/resources, src/test/java, src/test/resources.
  • You may delete src/main/* folders if not needed.

2.2 Adding Selenium WebDriver Dependency

Open pom.xml and add:

<dependencies>
    <!-- Selenium Java -->
    <dependency>
        <groupId>org.seleniumhq.selenium</groupId>
        <artifactId>selenium-java</artifactId>
        <version>4.18.1</version>
    </dependency>
</dependencies>

Update project: Right‑click project → Maven → Update Project. Maven downloads all required JARs.

2.3 Adding TestNG (Optional but Recommended)

<dependency>
    <groupId>org.testng</groupId>
    <artifactId>testng</artifactId>
    <version>7.9.0</version>
    <scope>test</scope>
</dependency>

Best Practice: Always keep your pom.xml clean and version numbers up‑to‑date.

3. Driver Executables and Browser Invocation

3.1 The Need for Driver Proxies

Browsers have strict security; Selenium cannot directly command them. Instead, each browser vendor provides a driver executable that acts as a bridge:

Your Java Code  →  Driver Executable  →  Browser
Chrome → chromedriver (or chromedriver.exe)
Firefox → geckodriver
Edge → msedgedriver

3.2 Two Ways to Provide the Driver

A. Manual Setup – System.setProperty() (Traditional)
Download the matching driver, extract to a folder, set property:

System.setProperty("webdriver.chrome.driver", "C:\\drivers\\chromedriver.exe");
WebDriver driver = new ChromeDriver();

Drawbacks: Manual version matching, updates manual, path errors common.

B. Automatic Management – Selenium Manager (Modern, Recommended)
From Selenium 4.6.0+, Selenium includes Selenium Manager. It detects your installed browser version, automatically downloads the correct driver, caches it, and uses it.

WebDriver driver = new ChromeDriver();  // No property setting needed!

This is the preferred approach for most development and CI/CD environments. When to fall back to manual? When the machine has no internet access or you must pin a specific driver version.

3.3 macOS Security Workaround

Right‑click (or Control‑click) the driver file → Open. In the dialog, click Open to grant an exception. Alternatively, terminal: xattr -d com.apple.quarantine /path/to/driver.

4. Writing Your First Automation Test

4.1 Creating the Test Class

Inside src/test/java, create a class SeleniumIntroduction with main method.

import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;

public class SeleniumIntroduction {
    public static void main(String[] args) {
        WebDriver driver = new ChromeDriver();
        driver.get("https://rahulshettyacademy.com");
        String pageTitle = driver.getTitle();
        System.out.println("Page Title: " + pageTitle);
        String currentUrl = driver.getCurrentUrl();
        System.out.println("Current URL: " + currentUrl);
        driver.quit();
    }
}

4.3 Understanding Each Method

  • driver.get("URL") – Navigates to the specified URL.
  • driver.getTitle() – Returns the page title as a String.
  • driver.getCurrentUrl() – Returns the exact URL the browser is currently displaying.
  • driver.close() – Closes the current browser window/tab that WebDriver is focused on.
  • driver.quit() – Closes all windows opened by the WebDriver session and shuts down the driver process.

4.4 close() vs quit() – Critical Interview Question

Featuredriver.close()driver.quit()
ScopeCloses only the current focused windowCloses all windows/tabs
SessionDoes not end the WebDriver sessionEnds the session completely
When to useSingle‑window scripts (rare)Always preferred; prevents hanging processes

Automation Engineer’s rule: Always use driver.quit() unless you have a very specific reason to keep other windows open. It ensures a clean test teardown.

4.5 Best Practice: Declare Driver as WebDriver type

WebDriver driver = new ChromeDriver();      // Recommended (program to interface)

Using WebDriver as the reference type limits visible methods to those defined in the WebDriver interface, making your code easily portable to other browsers.

5. Cross‑Browser Testing – One Code, Many Browsers

The real power of Selenium lies in writing a test once and running it on different browsers with minimal changes.

5.1 Running on Firefox

WebDriver driver = new FirefoxDriver();
driver.get("https://rahulshettyacademy.com");
System.out.println(driver.getTitle());
driver.quit();

5.2 Running on Microsoft Edge

WebDriver driver = new EdgeDriver();
driver.get("https://rahulshettyacademy.com");
System.out.println(driver.getTitle());
driver.quit();

5.3 Running on All Three Browsers (Example with reusable method)

public class CrossBrowserDemo {
    public static void main(String[] args) {
        WebDriver driver;
        driver = new ChromeDriver(); runTest(driver, "Chrome");
        driver = new FirefoxDriver(); runTest(driver, "Firefox");
        driver = new EdgeDriver(); runTest(driver, "Edge");
    }
    public static void runTest(WebDriver driver, String browser) {
        driver.get("https://rahulshettyacademy.com");
        System.out.println(browser + " - Title: " + driver.getTitle());
        driver.quit();
    }
}

5.4 A Note on Internet Explorer

Internet Explorer (IE) is deprecated. Selenium’s InternetExplorerDriver is unreliable and slow. Always use Microsoft Edge for Microsoft‑based automation. Modern enterprise automation targets Chrome, Firefox, and Edge only.

6. Practical Insights for Automation Engineers, Testers & Developers

6.1 Summary of Key Interview Questions

  • What is Selenium WebDriver? Answer covering the three definitions: a Selenium component, a Java interface, and an API.
  • Why is WebDriver designed as an interface? To enforce a common contract across all browsers, enabling cross‑browser testing without code changes.
  • Why do we write WebDriver driver = new ChromeDriver(); instead of ChromeDriver driver = new ChromeDriver();? It follows the “program to interface” principle, hides browser‑specific methods, and makes the code browser‑independent.
  • Difference between close() and quit()? close() shuts the current window; quit() shuts all windows and terminates the session.
  • What is the role of a driver executable (e.g., chromedriver)? It acts as a proxy between Selenium code and the browser, translating commands.
  • What is Selenium Manager? A built‑in tool (since Selenium 4.6.0) that automatically downloads and manages browser drivers, removing the need for manual System.setProperty().

6.2 Developer & Tester Best Practices

  • Maven always: Never download JARs manually.
  • Use Selenium Manager: Simplifies setup and avoids version mismatch headaches.
  • driver.quit() in teardown: Always quit the driver at the end of a test (or in @AfterMethod when using TestNG) to avoid resource leaks.
  • Print intermediate values: Use System.out.println to debug page titles, URLs, and element attributes.
  • Keep the same code for all browsers: Leverage polymorphism; avoid browser‑specific code paths.
  • Stay updated: Selenium evolves rapidly. Check Selenium.dev for the latest version.
  • Assertions are next: Replace if statements with proper assertion libraries (TestNG or JUnit) in real frameworks.

6.3 What’s Coming Next?

Locating Web Elements: Using ID, name, class, CSS selectors, XPath. Performing Actions: click(), sendKeys(), dropdown handling, waits. Building a robust framework: Page Object Model, TestNG integration, reporting, data‑driven tests.

7. Complete Setup Checklist for a New Automation Project

  • ✅ Install Java JDK (8, 11, or 17) and Eclipse/IntelliJ.
  • ✅ Create a Maven project in the IDE.
  • ✅ Add selenium-java dependency in pom.xml (latest version).
  • ✅ Add TestNG dependency (optional, for later use).
  • ✅ Update Maven project to download all JARs.
  • ✅ Install browsers (Chrome, Firefox, Edge).
  • ✅ Write the first test using WebDriver driver = new ChromeDriver(); (Selenium Manager handles the driver automatically).
  • ✅ Execute the test; verify the browser opens, navigates, and closes.
  • ✅ Extend the script to capture title and URL, and implement driver.quit() for cleanup.
